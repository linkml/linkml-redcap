# REDCap Data Dictionary as a LinkML Schema

## Formalising REDCap's Structure for Ontology-Based Instrument Generation

**Author:** Adam Graefe, BIH-CEI (Berlin Institute of Health, Center for Digital Health)  
**Date:** April 2026  
**Status:** Early-stage proposal — seeking community input

---

## The Problem

REDCap is the most widely used electronic data capture system in rare disease research, but its data dictionaries are essentially unstructured CSV files with implicit rules. When building **ontology-based** instruments — where variable names encode ontology codes, choices map to coded terms, and annotations carry semantic metadata — the process is entirely manual and error-prone.

The [RareLink framework](https://github.com/BIH-CEI/rarelink) demonstrates what's possible: a full ontology-based rare disease common data model embedded in REDCap with preconfigured export to GA4GH Phenopackets and HL7 FHIR. But as demand grows for disease-specific extensions (e.g., the Canadian Inborn Errors of Immunity National Registry), we face a scalability bottleneck: every new instrument requires hand-crafting a REDCap CSV that adheres to both REDCap's own rules *and* the ontology-based conventions that enable downstream interoperability.

**What if we could formalise these rules as a LinkML schema?**

---

## The Proposal

Create a **LinkML meta-schema** (`redcap-data-dictionary`) that models the structure of REDCap data dictionaries themselves — not a specific data model, but the data model *of* data dictionaries. This schema captures:

### 1. REDCap's Native Rules (18 CSV columns)

Every REDCap data dictionary is a CSV with exactly 18 columns. The schema models each column as a slot with its constraints:

- **`variable_field_name`**: lowercase, underscores, unique, cannot start with a number → `pattern: "^[a-z][a-z0-9_]*$"`
- **`form_name`**: same constraints, fields must be contiguous by form
- **`field_type`**: constrained enum (`text`, `dropdown`, `radio`, `checkbox`, `yesno`, `calc`, `file`, `slider`, `descriptive`, etc.)
- **`choices_calculations_slider_labels`**: pipe-delimited `raw_value, label` pairs for categoricals; calculation expressions for calc fields; `BIOPORTAL:ONTOLOGY` for autocomplete
- **`text_validation_type`**: `date_ymd`, `integer`, `number`, `email`, etc.
- **`branching_logic`**: REDCap's conditional display syntax
- **`field_annotation`**: free text, but in ontology-based models carries structured metadata

### 2. Ontology-Based Conventions (RareLink patterns)

Layered on top of the base REDCap rules, the schema captures the patterns that make ontology-based instruments work:

**Variable Naming Convention:**
```
{ontology_prefix}_{concept_id}[_{qualifier}]
```
Examples: `snomedct_184099003` (Date of Birth), `loinc_76689_9` (Sex at Birth), `hp_0012823_hp1` (HPO modifier 1)

**Choice Encoding:**
```
{ontology_prefix}_{concept_id}, {Human Label}
```
Example: `snomedct_248152002, Female | snomedct_248153007, Male`

**BioPortal Autocomplete:**
```
BIOPORTAL:{ONTOLOGY_NAME}
```
Example: `BIOPORTAL:HP` for HPO term search, `BIOPORTAL:MONDO` for disease coding

**Structured Annotation Format:**
```
Variable:
ONTOLOGY:CODE | Label |
Choices:
- ONTOLOGY:CODE | Label
Version(s):
- Ontology Name VERSION
Mapping:
- HL7 FHIR Expression v4.0.1: Resource.element
- GA4GH Phenopacket Schema v2.0 Element: Block.field
```

### 3. Instrument-Level Rules

- Form naming: `{project_prefix}_{N}_{section_name}` (e.g., `rarelink_5_disease`)
- Completion status field auto-generated by REDCap per instrument
- Repeating vs. non-repeating instrument designation
- First field must be the record identifier

## What the Schema Enables

### For the LinkML Community
- **A reference model for REDCap integration**: any LinkML-based project targeting REDCap can use this schema to validate their data dictionary output
- **LinkML-Map bridge**: transformation specifications from arbitrary LinkML schemas → REDCap DD CSV, with the meta-schema as the target model
- **Schemasheets compatibility**: the meta-schema could inform a Schemasheets-based workflow for authoring REDCap instruments in spreadsheets

### For the Rare Disease Community
- **Scalable instrument development**: disease-specific extensions no longer require hand-crafting CSVs — input ontology codes, get a valid instrument
- **Built-in interoperability**: generated instruments carry FHIR and Phenopacket mappings from the start
- **Automated validation**: catch REDCap rule violations (variable naming, choice format, branching syntax) before upload
- **LLM-assisted generation**: the schema serves as a contract for a Claude skill that can generate instruments from natural language descriptions plus ontology codes

### For the RareLink Project
- **Simplified extensional workflow**: new disease-specific instruments become a configuration task, not a development task (addresses [#223](https://github.com/BIH-CEI/rarelink/issues/223))
- **Improved mapping structure**: YAML-based mappings generated alongside instruments (addresses [#204](https://github.com/BIH-CEI/rarelink/issues/204))
- **LLM-based automation**: Claude skill for instrument creation with ontology validation (addresses [#204](https://github.com/BIH-CEI/rarelink/issues/204))

## Current State of the Schema

The initial `redcap_data_dictionary.yaml` schema (v0.1.0) includes:

**Classes:**
- `DataDictionary` — top-level container (tree_root)
- `Field` — models one CSV row with all 18 columns as typed/constrained attributes
- `Choice` — structured representation of a single permissible value
- `OntologyAnnotation` — structured field annotation for ontology-based models
- `Instrument` — logical grouping derived from form_name

**Enums:**
- `FieldType` — all 12 REDCap field types
- `TextValidationType` — all validation rules (date, number, email, etc.)
- `CustomAlignment`, `IdentifierStatus`, `MatrixRanking`
- `OntologyPrefix` — recognised ontology prefixes for variable naming
- `BioPortalOntology` — available BioPortal ontologies for autocomplete

## Open Questions for the Community

1. **Schema organisation**: should the ontology-based extensions (RareLink patterns) live in the same schema or as a separate import? The base REDCap meta-schema is useful independently.

2. **Validation depth**: how far should LinkML-native validation go vs. delegating to external validators? For example, branching logic syntax is complex — should we model it as a structured class or keep it as a string with external validation?

3. **LinkML-Map integration**: has anyone explored using LinkML-Map's `TransformationSpecification` to map between a domain schema and a flat CSV target like REDCap's DD format? The row-per-field structure is unusual for LinkML.

4. **Schemasheets bridge**: could a Schemasheets configuration be defined that maps spreadsheet columns → the meta-schema → REDCap DD CSV? This would give domain experts a spreadsheet-native authoring experience.

5. **Code generation targets**: beyond the CSV, should the schema support generating REDCap's XML project format (used for cross-project imports)?

6. **Existing work**: is anyone else working on formalising REDCap data dictionaries in LinkML or a similar modelling language?

## Real-World Validation

The schema is being validated against the production [RareLink-CDM v2.0.6 data dictionary](https://github.com/BIH-CEI/rarelink) (108 fields, 11 instruments, 21 ontologies) and the CIEINR extension. The RareLink framework has been deployed across four countries (Germany, Canada, South Africa, Japan) covering diverse rare disease domains.
